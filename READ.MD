## 2. 목표 서비스 및 실제 구현 정도

### 🎯 목표 서비스

- 사용자가 **의약품 이름을 검색**하면 공공데이터 API를 통해 정보를 제공

- 사용자가 **증상을 선택**하면 관련 의약품을 추천

- 추천 로직을 단순 나열이 아닌 **데이터 관계 기반**으로 구현

## 3. 단계별 구현 과정 및 학습 내용

### ① 기획 및 설계 단계

- 서비스 도메인을 **“의약품 정보 + 증상 추천”**으로 정의

- 단순 검색 서비스가 아닌 **추천 기능을 중심**으로 기획

- 데이터 흐름을 먼저 설계한 후 기능 구현에 착수

📌 **학습한 점**

- 기능 구현 전에 **데이터 관계 설계가 중요**하다는 점을 체감

---

### ② 백엔드 구현 단계 (Django + DRF)

#### 🔹 의약품 검색

- 공공데이터포털 **의약품개요정보(e약은요) API** 활용

- 검색 요청 시 외부 API 호출 → DB 저장 → 응답 반환

`검색 요청 → 외부 API → DB 저장 → 결과 반환`

📌 **어려웠던 점**

- 외부 API 응답 구조가 일정하지 않아 예외 처리 필요

- API Key 보안 관리 필요

📌 **새로 배운 점**

- `.env` 파일을 활용한 민감 정보 관리

- DRF에서 예외 처리 및 상태 코드 분리

---

### ③ 추천 기능 구현 단계

#### 🔹 증상 기반 추천 로직

- Symptom ↔ Drug **ManyToMany 관계** 설계

- 증상 선택 시 연결된 의약품 조회

📌 **기술적 핵심**

- 추천 알고리즘이 아닌 **데이터 관계 기반 추천**

- 하드코딩 없는 확장 가능한 구조

📌 **어려웠던 점**

- 추천이 “안 되는 것처럼 보였던” 원인이  
  → 실제로는 **연결된 데이터가 없었던 문제**

📌 **배운 점**

- 추천 시스템에서 **데이터 품질과 연결 관계가 핵심**임을 이해

---

### ④ 프론트엔드 구현 단계 (Vue 3)

- Composition API 기반 컴포넌트 설계

- Axios로 API 통신

- 카드형 UI로 검색 결과 및 추천 결과 출력

📌 **학습한 점**

- API 구조에 맞는 UI 설계의 중요성

- 비동기 처리와 상태 관리 흐름 이해

## 4. 데이터베이스 모델링 (ERD)

### 🔹 주요 테이블

- **Drug**
  
  - id
  
  - name
  
  - effect
  
  - usage
  
  - warning

- **Symptom**
  
  - id
  
  - name

- **관계**
  
  - Symptom ↔ Drug (ManyToMany)

📌 **설계 이유**

- 하나의 증상은 여러 의약품과 연결될 수 있고

- 하나의 의약품도 여러 증상에 사용될 수 있기 때문

---

## 5. 추천 알고리즘에 대한 기술적 설명

본 프로젝트의 추천 기능은 **규칙 기반 추천 방식**이다.

1. 사용자가 증상을 선택한다.

2. 선택한 증상(Symptom)을 조회한다.

3. 해당 증상과 연결된 의약품(Drug)을 조회한다.

4. 조회된 의약품 목록을 추천 결과로 반환한다.

`Symptom → ManyToMany → Drug`

이 구조는:

- 데이터가 늘어날수록 추천 품질이 향상되며

- 추가적인 알고리즘 없이도 확장 가능하다.
